# 基于 Uni-app 的校园二手交易平台 - 第二次项目进度汇报（第二周至当前）

## 一、阶段概览

本次汇报覆盖**第二周至当前阶段**（含联调期、功能补齐期与阶段性增强期）。阶段目标为：在第一周 MVP 骨架基础上，完成核心业务链路打通，并推进可演示能力从“能跑”提升到“可用、可测、可汇报”。

本阶段已落地的关键成果：
- 完成前后端全链路联调与真实登录鉴权。
- 完成图片上传、我发布的商品、管理端看板图表、接口文档自动化。
- 完成 IM（WebSocket）最小闭环，实现实时文本消息点对点通信。
- 补充关键服务单元测试，提升回归可控性。

## 三、成员阶段工作小结（四人逐一）

## 1. 肖家鑫（组长 / 后端架构师）

### 1）本阶段工作内容
- 设计并落地后端核心基础设施：跨域、分页、拦截器、静态资源映射。
- 推进认证体系改造：`AuthController` 对接数据库用户并将 token 存入 Redis。
- 推进数据库结构演进：用户角色字段、商品图片字段、测试数据维护。
- 推进接口文档能力：接入 SpringDoc 并配置文档访问白名单。
- 负责阶段技术路线把控与任务拆分协调。

### 2）遇到的问题以及解决思路
- 问题：前后端联调初期跨域与分页异常，影响联调节奏。  
  解决思路：补全 `CorsConfig` 与 MyBatis-Plus 分页插件，先打通基础设施再推进业务接口。
- 问题：接口鉴权边界不清，存在未登录可访问风险。  
  解决思路：统一由拦截器校验 token，并对游客接口采用白名单策略。
- 问题：文档与实现存在阶段性偏差。  
  解决思路：采用“源码即基线”策略，持续回写设计文档与阶段汇报。

### 3）下一阶段计划
- 完成订单状态机统一治理（取消、确认收货、状态流转约束）。
- 继续完善数据库索引与运维脚本，提升后续压测稳定性。
- 组织阶段回归验收与答辩材料技术口径统一。

---

## 2. 刘链凯（前端开发工程师 - 移动端）

### 1）本阶段工作内容
- 完成移动端登录页与登录态管理（token、本地用户信息）。
- 完成首页商品真实联调（搜索、分类筛选、详情跳转）。
- 完成发布页图片真实上传联调，修复描述字段绑定问题。
- 新增“我发布的商品”页面并接入 `/api/product/my`。
- 新增聊天页并打通详情页“聊一聊”入口。

### 2）遇到的问题以及解决思路
- 问题：前后端返回字段不一致（`message`/`msg`）导致提示异常。  
  解决思路：统一请求封装字段口径，错误提示集中处理。
- 问题：上传返回相对路径，页面显示出现不一致。  
  解决思路：在前端统一 URL 规范化，兼容绝对/相对路径。
- 问题：上传与发布存在并发时序问题（图片未上传完即提交）。  
  解决思路：增加“上传中禁止提交”和“至少一张图”校验。

### 3）下一阶段计划
- 开发“我买到的”页面并接入订单取消/确认收货操作。
- 优化聊天页体验（断线重连提示、历史消息加载、发送失败反馈）。
- 进行多端样式与交互回归，保障演示稳定性。

---

## 3. 陆道山（前端开发工程师 - 管理端）

### 1）本阶段工作内容
- 完成管理端 API 层封装与路由守卫。
- 完成用户管理页、商品审核页真实接口改造。
- 完成看板统计数据实时化与待审核列表联动。
- 引入 ECharts，新增用户增长与商品发布趋势图。

### 2）遇到的问题以及解决思路
- 问题：初期页面基于 Mock 数据，真实联调后字段和状态口径不统一。  
  解决思路：统一状态映射字典与接口参数封装，按后端字段重构表格展示逻辑。
- 问题：图表接入后打包体积上升。  
  解决思路：先保证可用性，后续通过按需加载和分包优化体积。
- 问题：用户状态筛选前后端联动不生效。  
  解决思路：推动后端补充 `status` 筛选，前后端共同闭环。

### 3）下一阶段计划
- 完善看板统计口径（近 7 天订单量、状态分布）。
- 补充管理端订单相关视图与运营筛选能力预留。
- 持续优化页面性能和交互一致性。

## 4. 陆际开（后端开发工程师 / 质量保证）

### 1）本阶段工作内容
- 完成 `ProductService` 分层改造，沉淀商品业务逻辑。
- 完成订单接口真实用户接入，移除硬编码用户 ID。
- 完成商品查询条件能力增强（关键词、分类、我的商品）。
- 完成 WebSocket IM 最小闭环（鉴权、点对点消息转发）。
- 完成关键单元测试编写（`ProductServiceImplTest`）。

### 2）遇到的问题以及解决思路
- 问题：高并发库存场景存在超卖风险。  
  解决思路：采用 Redis Lua 原子扣减并在数据库事务中落单。
- 问题：IM 长期停留在文档设计，缺少可演示实现。  
  解决思路：先做最小闭环（连接鉴权 + 文本消息），后续迭代持久化。
- 问题：服务层回归风险较高。  
  解决思路：优先补关键路径单测，覆盖映射和分页筛选核心逻辑。

### 3）下一阶段计划
- 落地 IM 消息持久化（会话表、消息表、历史查询）。
- 完成订单状态流转接口（取消、确认收货）及异常分支测试。
- 扩展接口自动化测试，构建稳定回归基线。

## 四、阶段问题与综合应对

1. 联调口径差异问题（字段、状态、返回结构）  
- 应对：统一 `Result` 结构与接口参数命名，更新文档基线。

2. 功能“有接口无页面”或“有页面无接口”问题  
- 应对：采用按链路验收（发布->审核->聊天->下单）的开发节奏，按链路逐段闭环。

3. 新功能迭代导致回归成本上升  
- 应对：补单测 + 固定回归清单 + 每周阶段汇报同步风险。
