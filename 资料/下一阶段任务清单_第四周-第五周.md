# 基于 Uni-app 的校园二手交易平台 - 下一阶段任务清单（第 4-5 周）

## 一、阶段目标

在当前“可演示 MVP + IM 最小闭环”基础上，完成订单闭环、IM 持久化、稳定性提升与验收材料沉淀，确保第 5 周末可进行完整业务链路演示与压力验证。

---

## 二、任务拆解（按优先级）

### P0（必须完成）

1. 订单闭环功能落地
- 后端新增订单取消、确认收货接口。
- 订单状态从“创建”扩展到“待交接 -> 已完成 / 已取消”。
- 移动端补“我买到的”页面并接入真实订单列表。
- 联动商品状态（确认收货后商品置为已售出）。

2. IM 能力增强（从“可连通”到“可用”）
- 新增 `chat_session`、`chat_message` 表结构与初始化脚本。
- 消息入库（发送时持久化）与历史消息查询接口。
- 聊天页进入时拉取历史消息，支持分页加载。

3. 接口文档与联调规范
- 基于 SpringDoc 完整校验核心接口分组与字段说明。
- 补充联调示例（登录、发布、聊天、下单、确认收货）。
- 输出一版可直接用于验收的 API 清单。

### P1（建议完成）

4. 管理端看板增强
- ECharts 新增“近 7 天订单量/成交量”趋势图。
- 统计口径统一（当天新增、累计总量、状态分布）。

5. 系统稳定性与异常处理
- 统一业务异常返回文案与错误码。
- 对 IM 断线重连、发送失败增加前端提示。
- 对订单重复提交增加防重校验（幂等键或短时锁）。

### P2（冲刺项）

6. 自动化测试补强
- 新增订单服务核心单测（确认收货、取消订单、状态非法流转）。
- 新增聊天消息服务单测（入库、拉取、会话过滤）。
- 增加关键接口集成测试脚本（Smoke Test）。

## 四、风险与应对

1. 风险：订单状态流转复杂，容易出现并发或越权更新。  
   应对：统一在 Service 层做状态机校验，关键接口加事务与鉴权校验。

2. 风险：聊天消息持久化后接口时延上升。  
   应对：会话按商品与双方维度建立索引，历史查询分页，必要时引入缓存。

3. 风险：阶段功能增多导致回归成本上升。  
   应对：按“发布-聊天-下单-收货”主链路建立固定回归用例并每日执行。

- 管理端可查看关键统计趋势和待处理数据。
- 接口文档齐全、核心单测可执行并通过。
